# d4-builder Data Spec

All data lives in `site/`. Paths below are relative to that directory. (This directory was previously named `workers/d4-builder/`; renamed 2026-04-09 because the project ships as a static Astro site, not a Cloudflare Worker.)

**This spec follows the optimizer.** Build composition comes from `data/optimizer_results.db` via `data/optimizer_v2.py`. Where the spec and the optimizer disagree, the spec gets updated to match the optimizer — never the other way around. The optimizer is the source of truth for what fields exist and what shape they take.

Hand-authoring of build composition is forbidden. See `~/.claude/projects/-Users-michael-code-d4builder/memory/feedback_no_fabricated_game_data.md` and `feedback_optimizer_is_source_of_truth.md`.

## Builds: 28 total, 4 per class

7 classes × 4 purposes = 28 builds. The 4 builds per class are differentiated by **purpose**, not by skill archetype:

| `purpose` | `purpose_label` | What it optimizes for |
|-----------|-----------------|----------------------|
| `pit` | Pit Push | Highest single-target / boss pressure for Pit pushing |
| `speed` | Speed Farm | Fast clears of low-density content |
| `leveling` | Leveling | Early-progression survival + steady damage growth |
| `mythic` | Mythic Unique | Endgame loadout assuming a mythic unique slot |

Build IDs are `<class_lower>_<purpose>` — e.g. `sorcerer_pit_push`, `barbarian_speed_farm`, `necromancer_leveling`. The 28-build set is generated by the optimizer; the frontend renders what's there.

## `builds/<id>.json` — top-level shape

The shape mirrors a row in `optimizer_results.db.optimizer_results`. Field names match the optimizer's column names so transcription is mechanical.

```jsonc
{
  "id": "sorcerer_pit_push",
  "build_name": "Sorcerer · Pit Push",
  "class": "Sorcerer",
  "purpose": "pit",
  "purpose_label": "Pit Push",
  "tier": "B",
  "build_score": 49079.216905,
  "global_rank": 15,
  "available": "live",
  "season": "Season 12 - Season of Slaughter",
  "patch": "2.6.0.70982",
  "specialization": "Enchantment Slots",
  "specialization_detail": { "name": "Enchantment Slots", "mechanic_type": "enchantment_slots" },

  // ── Skills ──────────────────────────────────────────────────────
  "skill_bar": ["Inferno", "Flame Shield", "Firewall", "Frozen Orb", "Ice Shards", "Lightning Spear"],
  "skill_bar_rank": 5,    // optimizer assumes active skills at rank 5
  "skill_upgrades": [
    { "name": "Inferno", "enhanced": "Prime Inferno", "morph": "Supreme Inferno" }
    // ... one per skill on the bar
  ],
  "passives": ["Align the Elements", "Elemental Synergies", /* ... 10 total */],
  "passive_rank": 3,      // optimizer assumes passives at rank 3
  "key_passive": "Combustion",

  // ── Gear ────────────────────────────────────────────────────────
  // Per-slot equipment. Each slot is a discriminated union: either a unique
  // (rarity:"unique", unique_id) or a legendary (rarity:"legendary",
  // base_item_type, aspect_id). NEVER a hand-written display name —
  // both unique display names AND legendary display names are computed at
  // render time from data/.
  "gear": {
    "Helm":    { "rarity": "unique", "unique_id": "Helm_Unique_Sorc_101", "_optimizer_name": "Hail of Verglas" },
    "Chest":   { "rarity": "unique", "unique_id": "Chest_Unique_Sorc_002", "_optimizer_name": "Raiment of the Infinite" },
    "Gloves":  { "rarity": "unique", "unique_id": "Gloves_Unique_Sorc_101", "_optimizer_name": "Flameweaver" },
    // ... one per slot
  },

  // ── Aspects (the optimizer's parallel field — see optimizer_concerns.md #2) ──
  // The optimizer also emits a per-slot list of aspect picks. Currently
  // structurally inconsistent with `gear` (every slot is a unique above, but
  // aspects exist for those same slots). The frontend renders this verbatim
  // as "Recommended legendary aspect alternatives" without trying to
  // reconcile. Slot names use Ring1/Ring2 (optimizer quirk).
  "aspects_recommended": [
    { "slot": "Helm", "aspect_name": "Accelerating", "max_value": 21.0 }
    // ...
  ],

  // ── Tempers (4-bucket map, not per-slot) ────────────────────────
  "tempers_recommended": {
    "offensive": "Natural Finesse",
    "defensive": "Pyromancy Endurance",
    "weapon": "Frost Augments",
    "mobility": "Natural Motion"
  },

  // ── Gems (3-bucket map: weapon/armor/jewelry) ───────────────────
  "gems_recommended": {
    "weapon": "Royal Emerald",
    "armor": "Royal Ruby",
    "jewelry": "Royal Skull"
  },

  // ── Paragon ─────────────────────────────────────────────────────
  // List of board+glyph picks. The traversal fields (activated_nodes,
  // points_spent, stats_gained, board_id) are populated by
  // paragon_pathfinder.plan_paragon at optimization time but are
  // currently empty for all rows — see optimizer_concerns.md #8.
  "paragon_boards": [
    { "board": "Icefall", "glyph": "Adept", "activated_nodes": [], "points_spent": 0, "stats_gained": {}, "board_id": "" }
    // ... up to 5 boards (D4 hard equip limit)
  ],

  // ── Runewords (Season mechanic) ─────────────────────────────────
  "rune_pair_1": { "ritual": "Feo", "invocation": "Lum", "score": 300.0 },
  "rune_pair_2": { "ritual": "Neo", "invocation": "Ton", "score": 36.0 },

  // ── Mercenary ───────────────────────────────────────────────────
  "mercenary": { "primary": "Aldkin", "reason": "..." },

  // ── Nightmare dungeons that drop the build's aspects ────────────
  "nightmare_dungeons": [
    { "name": "Pallid Delve", "zone": "Dry Steppes", "region": "Qara Yisu", "aspect": "Elementalist's", "x": -742.9, "y": -96.6 }
    // ...
  ],

  // ── Per-skill score breakdown (debug; renderer may hide) ────────
  "score_breakdown": {
    "Inferno": 2569.13, "Flame Shield": 7.05, /* ... */
  }
}
```

### Slot shape: unique vs. legendary

Per D4 mechanics (memory: `d4_legendary_gear_shape.md`), legendary item names are procedurally generated from the imprinted aspect's name template applied to the base item type. The build JSON reflects this:

**Unique slot:**
```json
{ "rarity": "unique", "unique_id": "Helm_Unique_Sorc_101" }
```

**Legendary slot:**
```json
{ "rarity": "legendary", "base_item_type": "Wand", "aspect_id": "<id from data/d4_stats.db.affixes>" }
```

The renderer computes the display name at render time:
- For uniques: looks up `unique_id` in `data/d4_stats.db.items` → `display_name`.
- For legendaries: looks up `aspect_id` in `data/d4_stats.db.affixes` (or `mr_affixes_extra` for prefix/suffix templates) → applies the prefix/suffix to `base_item_type`.

**Never store the display name.** Never hand-write a legendary item name anywhere. The `_optimizer_name` field on a unique slot is a traceability breadcrumb (what the optimizer originally said), not a display source.

The current optimizer's `select_gear()` only emits unique slots; legendary slots are documented in the spec for when the optimizer learns to emit them (or for future hand-curated builds, if any). See `optimizer_concerns.md #1`.

## What's NOT in this spec (intentionally dropped)

- **`variants[]` 5-stage progression** (leveling / early_legendary / full_legendary / legendary_uniques / mythic_uniques) — was speculative in the original spec, predates the optimizer, never matched optimizer output. Each build is now one state. If "how do I level into this build" content is wanted later, that's v2 with its own data source.
- **`stat_priority`** — the optimizer doesn't output one. The skill bar + tempers + paragon collectively imply the priority; rendering it as an explicit list would require either (a) a derivation function or (b) hand-authoring. Both are deferred to v2.
- **`runewords` (top-level array)** — replaced by `rune_pair_1` / `rune_pair_2` matching the optimizer's columns.
- **`gear` as `{weapon, helm, chest, ...}` named keys** — the optimizer uses PascalCase slot keys (`Helm`, `Chest`, `Gloves`, `Boots`, `Legs`, `Ring`, `Amulet`, `Offhand`, `Weapon`). The spec follows.
- **`math_justification`** — the optimizer's `score_breakdown` field carries this in numeric form.
- **`per-build guide markdown files`** — the optimizer doesn't produce them.

## Sourced provenance (still required)

Every numeric or factual claim that can be cited back to the optimizer or to `data/` SHOULD have an entry in the top-level `sources` object.

- **Shape:** flat object whose keys are dotted paths into the build JSON and whose values are human-readable provenance strings.
- **Format of the value (user-facing, not SQL):** `"<friendly source> · <what>"`. Example: `"optimizer_results.db · Sorcerer Pit Push · build_score 49079.22"`. The build-time loader may also store a `_debug_source` value with the raw SQL/file path for debugging.
- **Authoring rule:** never invent values. If a number is not in `data/optimizer_results.db` (or the source files the optimizer reads), it does not appear in the JSON.

## Data gap cards

If a section is intentionally incomplete, it MUST emit a `data_gap` block instead of being silently empty:

```json
"data_gap": {
  "reason": "human-readable explanation",
  "needs": ["specific thing 1", "specific thing 2"],
  "show_card": true
}
```

A `data_gap` MUST also be emitted (and console-logged) if a build references any item whose ID matches `[PH]` placeholders, `TestLook*` fixtures, or any internal_name ending in `_NoPowers`. The deploy success metric is **zero visible data gap cards on the production build**.
